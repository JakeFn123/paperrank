# 检索与评分系统详解（V2.1）

> 本文档重点解释：检索怎么跑、`query_match_score` 与 `rerank_score` 怎么算、最终评分怎么合成、证据审查怎么做。

---

## 1. 总体执行链

1. Planner 产出 `ResearchPlan`（含 `intent_slots` 与 `sub_queries`）。  
2. Retriever 调 MCP 检索多源论文。  
3. 检索后处理：`dedupe -> recall pool (~100) -> rerank -> top_k<=30`。  
4. Scorer 做证据召回与五维评分。  
5. Synthesizer 生成中文结论并做 `evidence_audit`。

---

## 2. 两阶段检索（本轮核心增强）

代码入口：`app/tooling.py::search_via_mcp`

### 2.1 Stage A：高召回

- 目标召回池：`SEARCH_RECALL_POOL`（默认 100）。
- 根据 `source` 和 `sub_queries` 动态计算每个子查询的 `effective_limit`：

```text
dynamic_limit = ceil(recall_target / (subquery_count * source_multiplier))
effective_limit = clamp(max(per_query_limit, dynamic_limit), 1, 20)
```

说明：
- `source_multiplier=2` 当 `source=all`（语义学术库 + arXiv 双源）。
- MCP 服务端对单次 limit 有硬上限 20。
- 若某子查询 0 命中，系统会自动做一次“简化查询”重试（保留前 10 个核心词）。

### 2.2 Stage B：重排压缩

- 对去重后的 recall pool 调用 `app/rerank.py::rerank_papers`。
- 默认尝试 Cross-Encoder（`sentence-transformers`），失败自动回退 lexical。
- 最终保留 `max_papers`（硬限制 `<=30`）。

输出新增字段：
- `PaperRecord.rerank_score`

检索日志新增一行：
- `query=__postprocess__`
- 包含 `recall_target`, `recall_after_dedupe`, `rerank_backend`, `rerank_model`, `rerank_fallback`

## 3. query_match_score 计算（检索解释分）

代码位置：`app/tooling.py::dedupe_papers`

```text
query_match_score = concept_hit_count * 1.0 + relevance_score + lock_match_count * 1.5
```

含义：
1. `concept_hit_count`：命中概念组数量。  
2. `relevance_score`：词项覆盖 + 技术短语 bonus。  
3. `lock_match_count`：命中的人工锁定概念数量（高权重）。

用途：
- 解释“为什么召回到这篇论文”。
- 不直接决定最终排序（最终排序优先看 `rerank_score` + `total`）。

---

## 4. rerank_score 计算（重排分）

代码位置：`app/rerank.py`

### 4.1 Lexical 分数

- 对 query 与论文 `title+abstract+venue+authors` 做词项覆盖。
- 叠加少量技术短语 bonus。
- 归一化到 `[0,1]`。

### 4.2 Cross-Encoder 分数

- 默认模型：`cross-encoder/ms-marco-MiniLM-L-6-v2`。
- 输入 `(query, doc)` 对，得到原始相关性分。
- 归一化到 `[0,1]`。

### 4.3 融合公式

```text
if cross_encoder available:
    final = 0.75 * cross_encoder_norm + 0.25 * lexical_norm
else:
    final = lexical_norm

rerank_score = final * 100
```

## 5. 最终 total 评分（排序分）

代码位置：`app/llm.py::score_paper`, `app/tooling.py::compute_quality_signal`, `compute_complementarity`

```text
total = 0.35*content_relevance
      + 0.25*method_relevance
      + 0.15*timeliness
      + 0.15*quality_signal
      + 0.10*complementarity
```

- `content_relevance/method_relevance/timeliness`：LLM 评分。  
- `quality_signal/complementarity`：规则评分。

## 6. 证据审查 evidence_audit（本轮新增）

代码位置：`app/agentic/subagents.py::SynthesisSubAgent`

审查维度：
1. 关键结论条目是否含 `[P#]` 引用。  
2. 引用编号是否存在于本次候选论文集合。  
3. 被引用论文是否存在可用证据片段。

输出结构示例：

```json
{
  "passed": true,
  "confidence": "high",
  "has_any_citation": true,
  "cited_refs": ["P1", "P2"],
  "invalid_refs": [],
  "missing_citation_lines": [],
  "refs_without_evidence": []
}
```

若未完全通过：
- 自动在最终报告追加“证据对齐审查”段落。  
- `confidence` 会降级为 `medium/low`。

## 7. 参数说明（与效果直接相关）

| 参数 | 位置 | 默认 | 影响 |
|---|---|---|---|
| `max_papers` | UI/CLI | 30 | 最终候选上限（<=30） |
| `per_query_limit` | UI/CLI | 10(UI)/6(CLI) | 每个子查询基础召回阈值 |
| `SEARCH_RECALL_POOL` | `.env` | 100 | Stage A 目标召回池大小 |
| `RERANK_BACKEND` | `.env` | auto | `auto/cross_encoder/lexical` |
| `RERANK_MODEL` | `.env` | ms-marco MiniLM | Cross-Encoder 模型名 |
| `locked_concepts` | UI/CLI | 空 | 强制概念对齐 |
| `intent_slots_override` | UI/CLI/CLI-json | 空 | 直接控制 Planner 槽位 |

---

## 9. 自检清单

1. 运行后 `search_log` 是否有 `__postprocess__`。  
2. 论文列表中是否可见 `Rerank` 指标。  
3. 候选数量是否 `<=30` 且通常大于 8。  
4. 最终结果是否包含审查指标（UI 或 CLI 的 `evidence_audit`）。
