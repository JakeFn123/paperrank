# 检索与评分系统详解（代码对齐版）

本文档回答两个核心问题：

1. 项目运行时，论文检索到底是怎么执行的。
2. 综合评分系统每个分数如何计算、如何影响最终排序。

本文内容严格对齐当前代码实现（`app/` + `mcp_servers/`），不是概念性方案。

---

## 1. 总体执行链路

主入口：`PaperEvaluationAgent.run()`（`app/pipeline.py`）

完整链路按顺序执行：

1. **问题拆解**：`LLMGateway.plan_question()` 将用户问题转成 `ResearchPlan`（研究意图 + 子查询 + 隐含前提 + 澄清问题）。
2. **多子查询检索**：`search_via_mcp()` 逐个子查询调用本地 Tool Server 的 `search_papers` 工具。
3. **结果标准化与去重**：统一字段格式、合并重复论文、按引用数/年份排序并截断。
4. **证据构建（可选）**：前 `ingest_top_n` 篇尝试下载 PDF、切块、向量化并检索证据片段。
5. **五维评分**：LLM 打 3 维 + 规则算 2 维，按固定权重合成总分。
6. **综合回答生成**：按总分排序后，产出中文分析结论；论文元数据与证据片段保持英文原文。

---

## 2. 检索系统详细逻辑

## 2.1 输入与问题拆解

输入是完整研究问题（例如：
`在企业知识库问答中，reranker 是否显著提升 RAG 稳定性？`）。

系统调用 `build_decompose_prompt()`，要求模型返回 JSON：

- `research_intent`
- `sub_queries`
- `hidden_assumptions`
- `clarification_questions`

关键约束：

- **`sub_queries` 必须是英文技术短语**，提升学术检索召回率。
- 其他字段默认中文，符合“系统默认中文”的产品约束。

实现位置：

- `app/prompts.py` -> `build_decompose_prompt()`
- `app/llm.py` -> `plan_question()`

---

## 2.2 本地 Tool Server（MCP 风格）调用方式

主流程不直接请求 arXiv / Semantic Scholar，而是先起本地服务：

1. 随机分配本地端口。
2. 子进程启动 `mcp_servers/academic_search_server.py --port <port>`。
3. 轮询 `GET /health` 等待服务可用。
4. 通过 `POST /tools/call` 调用工具 `search_papers`。
5. 检索结束后自动终止子进程。

主流程侧实现：

- `app/tooling.py` -> `ToolServerClient`
- `app/tooling.py` -> `search_via_mcp()`

服务侧实现：

- `mcp_servers/academic_search_server.py`

---

## 2.3 `search_papers` 的多源检索策略

`search_papers(query, limit, source)` 支持三种来源：

- `source=semantic_scholar`
- `source=arxiv`
- `source=all`（并发双源）

### A. Semantic Scholar 逻辑

接口：`/graph/v1/paper/search`

请求字段：

- `paperId,title,abstract,year,venue,citationCount,authors,url,openAccessPdf`

关键机制：

1. **429 限流重试**：最多 3 次。
2. 优先读 `Retry-After`，否则按 `1.5 * (attempt+1)` 秒退避。
3. 非 429 或网络错误最终降级为空列表，不中断全流程。

### B. arXiv 逻辑

接口：`https://export.arxiv.org/api/query`

请求参数：

- `search_query`
- `max_results`
- `sortBy=relevance`
- `sortOrder=descending`

关键机制：

1. **强制 3 秒间隔**（全局锁 + 时间戳），避免超频请求。
2. `feedparser` 解析 Atom，提取：title、summary、authors、pdf_url、abstract_url、year。

### C. `source=all` 聚合

- 并发执行 Semantic Scholar + arXiv。
- 任一源失败不影响另一源。
- 最终返回两边结果拼接列表。

## 2.4 检索结果标准化、去重、截断

## 2.4.1 标准化（防止脏数据导致崩溃）

每条记录先过 `_normalize_paper_row()`：

- 文本字段 `None -> ""`（如 `abstract`）
- `citation_count -> int`（失败回 0）
- `year -> int | None`（异常回 `None`）
- `authors` 统一成字符串数组

这一步解决了你之前遇到的：

- `ValidationError: abstract Input should be a valid string`

## 2.4.2 去重合并规则

去重键优先级：

1. 优先使用 `paper_id`
2. 若缺失，用 `sha1(title + year + source)` 生成稳定 ID

重复论文合并策略：

1. 保留更长摘要（信息更多）
2. 如旧记录缺 `pdf_url`，则补齐
3. 如旧记录缺 `paper_url`，则补齐

## 2.4.3 排序与保留上限

- 去重后按 `(citation_count, year)` 降序排序。
- 最终截断到 `SEARCH_TOP_K`（默认 `8`）。

配置位置：`app/config.py`。

## 2.5 检索日志与可见结果

每个子查询都会生成一条日志：

- `query`
- `hits`
- `source`
- `status`（`ok` / `error`）
- `error`（仅失败时）

日志用于 UI 侧边栏“检索轨迹”展示，便于定位“为什么没搜到”。

## 3. 证据层（RAG）如何参与评分

并非所有论文都做全文分析，只有前 `ingest_top_n` 篇尝试入库。

流程：

1. 有 `pdf_url` 才下载 PDF 到 `data/pdfs/<paper_id>.pdf`。
2. 用 PyMuPDF (`fitz`) 按页提取文本。
3. 分块参数：`chunk_size=1300`，`overlap=180`。
4. 调用 embedding（OpenAI 或 mock）向量化。
5. 存入本地 JSON 向量库：`data/chroma/chunks.json`。
6. 评分前按问题向量做 cosine 检索，取每篇论文 top_k 证据。

若 PDF 不可用或解析失败，系统会回退使用摘要作为证据，保证流程不中断。

---

## 4. 评分系统详解（重点）

评分对象：每篇候选论文。

输出结构：`PaperScore`（`app/schemas.py`）

字段：

- `content_relevance`
- `method_relevance`
- `timeliness`
- `quality_signal`
- `complementarity`
- `total`
- `rationale`

所有维度标称范围 `0-100`。

---

## 4.1 三个 LLM 维度

由 `LLMGateway.score_paper()` 调模型返回 JSON：

1. **content_relevance**
- 论文是否直接回答用户问题。

2. **method_relevance**
- 论文方法/实验设定是否可迁移到用户场景。

3. **timeliness**
- 相对候选年份分布是否有时效价值。

模型输入不只标题摘要，还包含证据片段（来自 RAG 或摘要回退）和候选年份分布。

---

## 4.2 两个规则维度

## 4.2.1 `quality_signal`（元数据质量信号）

计算函数：`compute_quality_signal()`（`app/tooling.py`）

公式：

```text
citation_signal = min(100, citation_count / 300 * 100)
venue_signal    = 70 (venue 非 arxiv) 否则 50
author_signal   = min(100, 40 + 8 * 作者数量)

quality_signal  = 0.45 * citation_signal
                + 0.35 * venue_signal
                + 0.20 * author_signal
```

解释：

- 引用数体现传播影响力（有上限，防止爆表）。
- venue 给予非 arXiv 小幅先验加分。
- 作者数量作为弱信号，不主导结果。

## 4.2.2 `complementarity`（互补性）

计算函数：`compute_complementarity()`（`app/tooling.py`）

流程：

1. 对每篇论文取 `title + abstract`，分词后保留长度大于 3 的 token。
2. 计算该论文与其他论文的 Jaccard 相似度。
3. 取平均相似度 `mean_sim`。
4. 转换成互补性分数：

```text
complementarity = (1 - mean_sim) * 100
```

解释：

- 越“和别人不重复”，分越高。
- 如果文本太少，默认趋向中性值（约 50）。

---

## 4.3 总分加权公式

最终排序使用 `total`：

```text
total =
  0.35 * content_relevance
+ 0.25 * method_relevance
+ 0.15 * timeliness
+ 0.15 * quality_signal
+ 0.10 * complementarity
```

权重意义：

1. 先确保“回答问题本身”（内容 + 方法共 60%）。
2. 再平衡“是否新、是否可靠、是否互补”（40%）。

实现：`app/llm.py` -> `score_paper()`。

---

## 4.4 计算示例（可手工复核）

假设某论文维度分为：

- `content_relevance = 82`
- `method_relevance = 75`
- `timeliness = 68`
- `quality_signal = 61`
- `complementarity = 70`

则：

```text
total = 0.35*82 + 0.25*75 + 0.15*68 + 0.15*61 + 0.10*70
      = 28.7 + 18.75 + 10.2 + 9.15 + 7
      = 73.8
```

系统会保留到小数点后两位并排序。

---

## 5. 从评分到最终展示的映射

1. 先按 `score.total` 降序。
2. 重编号 `P1, P2, ...`。
3. 把证据的 `ref_id` 同步为该编号。
4. 综合回答提示词要求：
- 输出中文 Markdown
- 引用必须带 `[P1]` 或 `[P1 p.3]`
- **论文标题/摘要/作者保持英文原文，不翻译**

这就是你在 UI 主界面看到的：

- 中文综合结论
- 英文论文列表（`st.dataframe`）
- 带页码证据片段

## 6. 为什么有时“看不到理想论文列表”

常见原因和对应排查：

1. **外部 API 限流**（尤其 Semantic Scholar 429）
- 观察侧边栏检索日志是否出现 error。
- 可临时切 `source=arxiv`。

2. **子查询质量一般**
- 问题太泛会导致拆解词泛化。
- 在问题中补充场景、指标、时间范围。

3. **召回参数偏小**
- 增加 `--per-query-limit`。

4. **截断上限影响**
- 提高 `SEARCH_TOP_K`。

5. **PDF 缺失导致证据弱化**
- 增加 `ingest_top_n`，或优先有 `pdf_url` 的来源。

---

## 7. 可调参数与效果

| 参数 | 位置 | 作用 | 建议 |
|---|---|---|---|
| `SUBQUERY_COUNT` | `.env` | 子查询数量 | 3-5 |
| `per_query_limit` | CLI/UI | 每个子查询召回数 | 4-10 |
| `SEARCH_TOP_K` | `.env` | 去重后保留上限 | 8-20 |
| `ingest_top_n` | CLI/UI | 做全文证据的论文数 | 1-6 |
| `source` | CLI/UI | 数据源选择 | `all` / `arxiv` |

---

## 8. 当前评分系统边界（实话实说）

1. `complementarity` 基于词项 Jaccard，不是语义级去重。
2. `quality_signal` 未引入期刊分区、H-index、实验可复现指标。
3. 证据检索是“单论文内 top-k chunk”，未构建跨论文证据图。
4. 分歧检测主要由综合生成阶段完成，不是独立 NLI 判别器。

以上不影响当前版本可运行、可解释、可迭代。
