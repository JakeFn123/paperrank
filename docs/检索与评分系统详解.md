# 检索与评分系统详解（V2 Agentic 代码对齐版）

本文档回答 4 个关键问题：

1. 重构后检索链路具体怎么跑。
2. `tools` / `subagents` / `skills` 如何参与检索与评分。
3. `query_match_score` 与 `total` 两套分数如何计算。
4. 如何从 UI 与 task trace 验证每一步确实执行过。

> 对齐代码：`app/agentic/*.py`, `app/tooling.py`, `app/llm.py`, `app/rag.py`, `mcp_servers/academic_search_server.py`

---

## 1. 执行总览（重构后）

主入口仍是：`PaperEvaluationAgent.run()`，但内部已委托给：

- `PaperRankAgentLoop.run()`（`app/agentic/loop.py`）

主流程：

1. PlannerSubAgent：拆解问题 -> `ResearchPlan`
2. RetrievalSubAgent：调用 `search_batch` 工具 -> `papers + search_log`
3. ScoringSubAgent：RAG 证据 + 五维评分 -> `ScoredPaper[]`
4. SynthesisSubAgent：综合写作 -> `final_answer_markdown`
5. 输出附带：`task_board_snapshot + loop_trace`

---

## 2. 检索链路（Tooling + MCP）

## 2.1 检索由哪个 Tool 触发

工具定义位置：`app/agentic/tools.py`

检索工具：`search_batch`

参数：
- `question`
- `sub_queries`
- `source` (`all/arxiv/semantic_scholar`)
- `per_query_limit`
- `locked_concepts`

内部绑定：`app.tooling.search_via_mcp(...)`

---

## 2.2 `search_via_mcp` 具体做了什么

1. 启动 `ToolServerClient`（随机端口拉起本地 MCP-style 服务）。
2. 对每个 `sub_query` 调用 `search_papers`。
3. 汇总 `search_log`（query/hits/status/error）。
4. 收集所有原始候选后，调用 `dedupe_papers()` 做标准化、去重与重排。

返回：`(papers, search_log)`。

---

## 2.3 MCP 服务侧检索逻辑

服务文件：`mcp_servers/academic_search_server.py`

`search_papers` 支持：
- `source=semantic_scholar`
- `source=arxiv`
- `source=all`（并发双源）

关键点：

1. Semantic Scholar：
- 支持 API key。
- 429 自动退避重试（最多 3 次）。

2. arXiv：
- 请求前强制 3 秒间隔（全局锁）。
- 结构化提取 title/summary/year/authors/pdf_url。

3. locked_concepts：
- 在检索阶段注入查询（提高召回）。
- 严格约束主要在检索后排序与过滤阶段完成。

---

## 3. 检索后处理（去重 + 匹配）

位置：`app/tooling.py` -> `dedupe_papers()`

### 3.1 标准化

- 文本字段：`None -> ""`
- `citation_count -> int`
- `year -> int | None`
- `authors -> list[str]`

### 3.2 去重

主键优先：`paper_id`  
回退：`sha1(title + year + source)`

冲突合并规则：
1. 保留更长 abstract
2. 补齐缺失 `pdf_url`
3. 补齐缺失 `paper_url`

### 3.3 概念匹配与锁定概念

- 提取 anchors：`question + sub_queries + locked_concepts`
- 命中概念组：`matched_concepts`
- 统计锁定概念命中数：`lock_match_count`
- 自适应过滤阈值：
  - 1 个锁定概念 -> 至少命中 1
  - 2-3 个 -> 至少命中 1
  - 4+ 个 -> 至少命中 2

若严格过滤后为空，会回退到部分命中候选，避免零召回。

## 4. 两套分数体系

## 4.1 检索阶段：`query_match_score`

在 `dedupe_papers()` 中计算：

```text
query_match_score = concept_hit_count * 1.0 + relevance_score + lock_match_count * 1.5
```

组成：
1. `concept_hit_count`：命中的概念组数量。
2. `relevance_score`：词项覆盖 + 技术短语 bonus。
3. `lock_match_count`：命中的人工锁定概念数量（高权重）。

用途：
- 候选论文排序与筛选解释。
- 不用于最终论文总分排序。

---

## 4.2 最终阶段：`total`（综合评分）

ScoringSubAgent 对每篇论文调用 `score_single`（绑定 `LLMGateway.score_paper`）。

### 4.2.1 三个 LLM 维度

- `content_relevance`
- `method_relevance`
- `timeliness`

### 4.2.2 两个规则维度

1. `quality_signal`（`app/tooling.py`）

```text
citation_signal = min(100, citation_count / 300 * 100)
venue_signal = 70 (non-arXiv) else 50
author_signal = min(100, 40 + 8 * author_count)
quality_signal = 0.45*citation_signal + 0.35*venue_signal + 0.20*author_signal
```

2. `complementarity`（`app/tooling.py`）

- 用 title+abstract 计算两两 Jaccard 相似度
- `complementarity = (1 - mean_similarity) * 100`

### 4.2.3 总分公式

```text
total = 0.35*content_relevance
      + 0.25*method_relevance
      + 0.15*timeliness
      + 0.15*quality_signal
      + 0.10*complementarity
```

`ScoredPaper` 按 `total` 降序作为最终排序。

## 5. RAG 证据如何参与评分

位置：`app/rag.py` + `ScoringSubAgent`

流程：
1. 对前 `ingest_top_n` 篇调用 `ingest_pdf`。
2. 下载 PDF 至 `data/pdfs/`。
3. 按页提取文本并切块（默认 `chunk_size=1300`, `overlap=180`）。
4. 向量写入 `data/chroma/chunks.json`。
5. 按问题向量检索 top_k 证据。
6. 评分时将证据片段喂给 `score_single`。

兜底：若 PDF 不可用，则用摘要构造 evidence，保证流程不断。

## 6. Skills 在检索评分中的作用

skills 不直接计算分数，但约束各子代理行为标准。

| Skill | 影响环节 | 主要影响 |
|---|---|---|
| `query-decomposition` | Planner | 降低子查询跑偏，提升检索相关性 |
| `academic-retrieval` | Retriever | 提升召回质量与日志可追踪性 |
| `evidence-grading` | Scorer | 强化证据优先与可解释评分 |
| `synthesis` | Synthesizer | 规范引用化输出与不确定性说明 |

## 8. 参数对检索和评分的影响

| 参数 | 位置 | 影响 |
|---|---|---|
| `source` | UI/CLI | 数据源范围与稳定性 |
| `per_query_limit` | UI/CLI | 召回候选数量 |
| `ingest_top_n` | UI/CLI | 全文证据覆盖深度与耗时 |

## 9. 常见偏差与排查建议

1. 子查询跑偏  
- 提供更具体问题或手动锁定概念。  
- 检查 Planner 输出的 `research_intent` 与 `sub_queries`。

2. 结果为空  
- 降低锁定概念数量或放宽表达。  
- 提升 `per_query_limit`。  
- 先测 `source=arxiv`。

3. 分数看似异常  
- 对照 UI 的“评分过程明细”检查维度贡献。  
- 核查证据片段是否与标题一致（防止摘要兜底误导）。

---

## 10. 一句话总结

重构后的检索与评分系统已经从“黑盒式串行流程”升级为“可拆分、可追踪、可审计”的标准智能体执行链：检索有工具边界、评分有证据与公式、执行有任务与轨迹。
